## Learned User Preferences

- Use Bun for this repo: `bun add`, `bun run` for scripts, and `bunx` for ad-hoc CLIs (Convex, shadcn); avoid `npx`/`npm`.
- Build UI with shadcn/ui components first; do not hand-roll primitives when a shadcn component exists; keep `Button` `className` merge as shadcn ships (`cn(buttonVariants({ variant, size, className }))`); when overriding press translate on absolutely centered icon buttons, use the full `active:not-aria-[haspopup]:…` modifier chain (shorter `active:` overrides won't replace base `active:translate-y-px`).
- When implementing from an attached plan, do not edit the plan file itself.
- Prefer Next.js App Router server patterns: async RSC pages, async `params` as `Promise`, and route-level `loading.tsx`/`error.tsx`/`not-found.tsx`; cluster routes use `preloadQuery` + client `usePreloadedQuery` (clusters list page 2+ via imperative `useConvex().query()` because Convex has no paginated preload); wallet pages use `preloadQuery` for clusters/deposits, cached `getLiveSnapshot` when present, `fetchAction(getSnapshot)` only when missing, then `WalletDetailLive` live subscriptions.
- Keep `'use client'` limited to interactive islands (search forms, tabs, providers); fetch wallet data in the server page.
- Use the `frontend-design` skill for branding and distinctive UI work; design language is a premium dark fintech terminal (near-black/charcoal surfaces, pale mint primary, green/red only for PnL semantics, thin borders, minimal rounding, no gradients/glass); Geist sans + monospace for addresses, monetary values, and timestamps; wordmark UI italicizes "trace" in Hypertrace (Hyperliquid-style), metadata titles stay plain "Hypertrace".
- Positions table: encode long/short with green/red asset badges; no separate Side column; Leverage column stacks multiplier + Cross/Isolated margin; funding fee negative/red = paid, positive/green = received; open-summary bar (Long/Short/Total/Funding/uPnL) via `Table` `leading` (USD only); uPnL cell stacks USD + ROE % (`unrealizedPnl / marginUsed` via `lib/position-roe.ts`); sortable on Value/Funding fee/uPnL (default Value desc) via lightweight client-side sort on the existing `Table` (parse HL numeric strings), not shadcn Data Table + TanStack; wallet tab tables share client pagination (`TABLE_PAGE_SIZE` 20, `paginateItems` + `TablePagination`).
- Route `loading.tsx` skeletons should mirror the live page layout (reuse `AppShell`, `SiteHeader`, and table structure).
- Home page is a two-column investigate layout: `SiteHeader variant="compact"` (logo + `SiteNavPills`) with `HomeHero` (badge/headline/feature cards) beside `InvestigationConsole` (`WalletSearch` autofocus); ⌘K focuses the wallet input via shadcn `Kbd` (no command palette); headline signature “move.” uses light italic Cormorant (~1.45em); desktop body vertically centered with slight upward bias and natural column heights (`items-center`, not forced equal stretch); recent chips only (no curated demos); omit fabricated home stats and cheesy protocol footers; in-input lucide `Search` icon submit; form `Input`s keep mobile `text-base` (16px) floor for iOS Safari focus zoom (`md:text-sm` ok on desktop)—do not wipe the floor via `twMerge`; prefer preset chips over free-text filters when a larger input looks out of place (e.g. leaderboard min account value); keep leaderboard filter/sort in URL search params (`LeaderboardView` in `lib/leaderboard-list.ts`) so back from a wallet restores the list; omit marketing aggregates (tracked equity, N-of-M) until real Convex sources exist.
- Wallet tabs on mobile: full-width equal segments (`flex-1` triggers, `w-full` list); desktop keeps content-sized tabs (`sm:flex-none`, `sm:w-fit`); Transfers tab label shortens to `Xfer` on mobile; sync tab + Positions/Spot sort to URL search params via `WalletView` (`lib/wallet-view.ts`) so back-navigation restores the view.
- Auth chrome (`AuthNavControls`): Sign in matches `SiteNavPills` text/link styling (not a heavy filled Button); signed-in state is Google profile avatar only—no display name in the header or sheet.
- Use `/poteto-mode` for nontrivial implementation, investigation, and architecture work; prefer free static denylists/labels for funding counterparties over paid Allium enrichment.

- Live pipeline: Hyperliquid WS → `worker/` (Railway, Bun) → Convex HTTP ingest (`/ingest/watches`, `/ingest/snapshot`, `/ingest/snapshot-timestamps`, `/ingest/leaderboard`) → `watchedAddresses` (24h TTL) + `walletSnapshots` (by address) + `leaderboardSnapshots` → client `useQuery`; one multiplexed WS connection per worker process (keep Railway `invigorating-sparkle` at 1 replica); `HL_WS_MAX_USERS` defaults to 10 and is hard-clamped at 10 to match HL’s unique-user WS subscription cap (not connection count)—WS + watch-sync cover only that newest hot set (deposit scans still cover all active watches); worker poll refreshes only stale snapshots (default 300s), caches shared perp + spot `metaAndAssetCtxs` (30s TTL, singleflight on miss), and routes all HL Info API POSTs through a global rate-limited queue; `upsertSnapshot` heartbeats `fetchedAt` when trading fields match (avoids rewriting identical snapshot docs); do not subscribe to Hyperliquid from the browser.
- Leaderboard ingestion (project Spec M4; README “M4” is deposit-source clustering — numbering differs): worker polls undocumented `https://stats-data.hyperliquid.xyz/Mainnet/leaderboard` (optional `LEADERBOARD_POLL_MS`, default 30m) and POSTs batches to `/ingest/leaderboard`; upserts `leaderboardSnapshots` + `wallets` only (never `watchedAddresses`); Convex actions cannot own this poll (V8 OOM / cloud cannot reach the stats host), so there is no leaderboard Convex cron/action fetch.
- Worker env: `CONVEX_URL` (prod `wonderful-condor-424.convex.cloud`), `WORKER_INGEST_SECRET` (must match Convex env), required `ARBITRUM_RPC_URL` (Alchemy Hypertrace app `lzbt77vxhe1kq3j6`; use `arb-mainnet` as Alchemy MCP network id), optional `HL_WS_URL`, `HL_WS_MAX_USERS` (default 10), `WATCH_POLL_MS`, `REFRESH_DEBOUNCE_MS`, `SNAPSHOT_STALE_MS` (default 300s / 5m), `WS_REFRESH_MIN_INTERVAL_MS` (default 5m), `META_CACHE_TTL_MS` (default 30s), `HL_MAX_CONCURRENCY` (default 3), `HL_MIN_REQUEST_INTERVAL_MS` (default 100), `LEADERBOARD_POLL_MS` (default 30m), `BRIDGE2_START_BLOCK` (default June 1 2026 block `468748168`), `FUNDING_LOOKBACK_DAYS` (default 7), `CCTP_EXTENSION_ADDRESS`, `CCTP_TOKEN_MESSENGER_V2` (Circle defaults on Arbitrum).
- Arbitrum transfer tracing (M3/M3.1): per watched wallet, scan USDC deposits to Bridge2/CCTP plus withdrawals (`FinalizedWithdrawal` + reverse CCTP Extension); rows live in `deposits` with `direction` (`deposit`|`withdrawal`); first scan backfills from `bridge2StartBlock` (not watch time); resolve upstream funding via Alchemy `getAssetTransfers` with `FUNDING_LOOKBACK_DAYS` before each deposit timestamp—amount-weighted among non-denylisted inbounds into primary `sourceAddress` + optional `funders[]` (backfill via `/ingest/deposits-missing-funders` on watch poll); denylist source of truth `data/funding-denylist.json` synced by `bun run sync:funding-denylist` into worker + Convex known-address modules; `internal.deposits.resetAllScanCursors` one-time backfill after scanner changes (deploy new worker first, then reset cursors); `sourceAddress === hlAddress` is often correct (self-funded, lookback miss, or all-infra skip), not necessarily a bug.
- Deposit-source clustering (README M4): signal 1 of 5; behavioral HL signals are the planned differentiator; only deposit-direction rows; skip denylisted/self sources and cap fan-out at 25; multi-funder edges use `funders[]` with weight ≥ 15% (`MIN_CLUSTER_FUNDER_WEIGHT`); `confidenceScore = min(1, 0.5 + 0.1 × (memberCount − 1))` in `depositClustering.ts` (member count only, basis `shared_deposit_source`); no Allium/`addressLabels` for Counterparty (prefer free static denylist); `clusters` table with `memberCount`; cron every 3 min runs `internal.clusters.rebuildDepositSource` using `.collect()` (Convex allows only one `.paginate()` per function); `clusterKey` format `deposit-source:0x…`; `clusters.list` paginated (20/page, `by_memberCount`); `getForWallet` uses indexed deposit-source lookup; `deposits.listByWallet` returns `{ deposits, hasMore }` capped at 100.
